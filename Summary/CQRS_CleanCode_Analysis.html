CastLibrary Backend Architecture Review

📊 Executive Summary

The CastLibrary backend demonstrates a solid attempt at CQRS pattern implementation with clear separation between commands and queries. However, there are several areas where the architecture deviates from best practices in clean code principles and CQRS purity. This analysis identifies 8 major issue categories with 28+ specific violations.

🚨 Issues Identified

1. Bloated Controllers with Too Many Dependencies 4 instances
CampaignsController: Constructor Injection Hell
CRITICAL
The CampaignsController has 35+ constructor dependencies including 31+ handler interfaces. This violates the Single Responsibility Principle and makes testing extremely difficult.
File: backend/CastLibrary.WebHost/Controllers/CampaignsController.cs (lines 18-57)
public class CampaignsController( IGetCampaignLibraryQueryHandler getLibraryQuery, IGetCampaignDetailQueryHandler getDetailQuery, ICreateCampaignCommandHandler createCommand, IUpdateCampaignCommandHandler updateCampaignCommand, // ... 31 more handlers ... IToggleShopItemScratchCommandHandler toggleShopItemScratchCommand, ICampaignWebMapper campaignMapper, IUserRetriever userRetriever, IHubContext<CampaignHub> hubContext) : ControllerBase
Recommendation: Introduce a CQRS Dispatcher/Mediator pattern to handle all command and query routing. This would reduce dependencies from 35+ to just 3-4 (Mediator, Mapper, UserRetriever, Hub).
Missing Mediator Pattern Implementation
HIGH
The project has individual handler interfaces (ICommandHandler, IQueryHandler) but lacks a central mediator to dispatch to them. Controllers directly reference all handlers.
Pattern Observed: Direct handler injection instead of mediator pattern
// Current (Anti-Pattern) var cast = await getCastLibraryQuery.HandleAsync(dmUserId); // Ideal (with MediatR or similar) var cast = await mediator.Send(new GetCastLibraryQuery(dmUserId));
Recommendation: Implement MediatR NuGet package or create a custom IMediator interface to centralize request/response routing.
2. Mixed Concerns in Handlers 5 instances
Image Processing Logic Mixed with Query Logic
HIGH
The GetCastLibraryQueryHandler mixes data retrieval with image URL generation. This violates SRP - a query handler should only fetch data.
File: backend/CastLibrary.Logic/Queries/Cast/GetCastLibraryQueryHandler.cs (lines 22-27)
public async Task<List<CastDomain>> HandleAsync(Guid dmUserId) { var cast = await castReadRepository.GetAllByDmAsync(dmUserId); var options = new ParallelOptions() { MaxDegreeOfParallelism = 4 }; // Image processing logic shouldn't be here! Parallel.ForEach(cast, options, member => { var imageKey = imageKeyCreator.Create(...); member.ImageUrl = imageStorageOperator.GetPublicUrl(imageKey); }); return cast; }
Recommendation: Create a separate ImageUrlEnricher service or move this logic to a post-processor/mapper layer that runs after the query.

3. CQRS Pattern Violations 4 instances
Commands Modifying Multiple Aggregates
HIGH
CreateCampaignCommandHandler creates a campaign AND inserts location instances in a single command. This violates CQRS - commands should have single, focused responsibility.
File: backend/CastLibrary.Logic/Commands/Campaign/CreateCampaignCommandHandler.cs (lines 19-34)
public async Task<CampaignDomain> HandleAsync(CreateCampaignCommand command) { var campaign = campaignFactory.Create(command.Request, command.DmUserId); var saved = await campaignRepository.InsertAsync(campaign); // Mixing concerns: also handling location instances int sortOrder = 0; foreach (var LocationId in command.Request.LocationIds) { var location = await locationReadRepository.GetByIdAsync(LocationId); var instance = locationInstanceFactory.Create(location, saved.Id, sortOrder++); await campaignRepository.InsertLocationInstanceAsync(instance); } return saved; }
Recommendation: Split into separate commands: CreateCampaignCommand and AddLocationsToCampaignCommand, or use Domain Events for eventual consistency.
Queries with Side Effects (Parallel.ForEach Mutation)
HIGH
Query handlers mutate the returned entities with image URLs. While not a database operation, this violates CQRS purity where queries should have no side effects.
File: backend/CastLibrary.Logic/Queries/Cast/GetCastLibraryQueryHandler.cs (lines 22-27)
Recommendation: Return DTOs with pre-calculated URLs instead of mutating domain objects in parallel.
Repository Pattern Mixed with CQRS
MEDIUM
Project uses separate read/write repositories (IxxxxReadRepository, IxxxxInsertRepository) but doesn't have clear command/query separation at the handler level.
Recommendation: Clearly separate read repositories (for queries) and write repositories (for commands) with distinct interfaces and implementations.
4. Scattered Validation Logic 6 instances
Validation in Controllers Instead of Handlers
HIGH
Controllers instantiate and run validators manually instead of centralizing validation in the command/query handler layer. This violates clean architecture principles.

Recommendation: Move validation into command handlers or use FluentValidation behaviors/pipeline behaviors. Controllers should be thin.
Multiple Validator Instantiation
MEDIUM
Validators are created new with new CreateCastRequestValidator() in controllers instead of being injected. This violates DI principles and makes testing difficult.
File: backend/CastLibrary.WebHost/Controllers/*.cs (multiple instances)
Recommendation: Register validators in DI container and inject them, or better yet, move to handler layer where validation is automatic.
5. Repository Layer Design Issues 3 instances
Read Repository Type Inconsistency
MEDIUM
Different repositories return inconsistent types. Some return Domain objects, some return DTOs, creating confusion about data flow.
File: backend/CastLibrary.Repository/Repositories/Read/ (multiple files)
Recommendation: Establish clear contract: read repositories return Domain objects, queries can map to DTOs if needed.
Null Reference Patterns
MEDIUM
Repositories and handlers return null for "not found" scenarios. This is problematic and should use Result patterns.
File: backend/CastLibrary.Logic/Commands/Auth/LoginCommandHandler.cs (line 20)
public async Task<AuthResponse> HandleAsync(LoginCommand command) { var user = await userReadRepository.GetByEmailAsync(command.Request.Email); if (user is null) return null; // Returns null instead of Result<T> // ... }
Recommendation: Implement a Result<T> type or use language features like IAsyncEnumerable with empty result handling.
6. IoC/DI Configuration Problems 3 instances
Massive IoC Registration Files
HIGH
IOCLogic.cs registers 70+ handlers with individual extension methods for Commands, Queries, Services, and Factories. This violates SRP and makes maintenance difficult.
File: backend/CastLibrary.WebHost/IoC/IOCLogic.cs (126+ lines)
Suggested approach: Auto-registration using reflection or factory patterns.
Recommendation: Use Scrutor for auto-registration of handlers by convention, reducing boilerplate from 200+ lines to 10-15 lines.
Scattered IoC Configuration Across Multiple Files
MEDIUM
IoC configuration split across 5 files (IOCLogic, IOCRepository, IOCAdapter, IOCWebhost, IOCRootContainer) makes it hard to see full dependency graph.
Files: backend/CastLibrary.WebHost/IoC/*.cs
Recommendation: Consider consolidating or creating a visual dependency diagram. Use a modular registration pattern.
7. Limited Testability 3 instances
Controllers Hard to Unit Test
HIGH
With 35+ dependencies in CampaignsController, creating meaningful unit tests would require mocking 35+ services. This is a test smell.
Recommendation: Introduce mediator pattern to reduce controller to handling framework concerns only (routing, auth, serialization).
No Command/Query Builder Pattern
MEDIUM
Commands are created with positional constructor parameters, making test data setup verbose and error-prone.
new CreateCampaignCommand(userId, request) // vs new CreateCampaignCommandBuilder() .WithUserId(userId) .WithRequest(request) .Build()
Recommendation: Create builder classes or use object initializers for commands/queries.
8. Architecture Documentation & Design Patterns 2 instances
No Clear CQRS Boundary Definition
MEDIUM
While Commands and Queries are separated, there's no clear documentation or enforcement of CQRS principles. Handlers can do both reading and writing without clear distinction.
Recommendation: Document CQRS rules: Commands = void-returning or ID-returning, Queries = data-returning, never both in same handler.
Missing Domain Events Pattern
MEDIUM
Complex operations that should trigger domain events (like campaign creation) have no event publishing mechanism visible.
Recommendation: Implement IDomainEvent pattern for publishing events from aggregates and handling them through separate event handlers.

🎯 Priority Recommendations

    • Move validation from controllers to command handlers using FluentValidation pipeline behaviors
    • Auto-register handlers using Scrutor to eliminate manual IoC registration
  2. Short-term (3-4 sprints):
    • Extract image enrichment from query handlers into separate service layer
    • Remove correlation and logging concerns from repository layer
    • Implement Result<T> pattern to eliminate null reference propagation
  3. Medium-term (5-6 sprints):
    • Split multi-operation commands into separate single-operation commands
    • Implement domain events for eventual consistency
    • Create comprehensive CQRS architecture documentation and patterns guide
