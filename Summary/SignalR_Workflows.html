⚡ SignalR Real-Time Workflows

What is SignalR?

SignalR is a real-time communication framework for ASP.NET Core that enables bidirectional communication between server and client. In the Cast Library application, SignalR is used to broadcast live updates to all connected users in a campaign when events occur (visibility changes, secrets revealed, time updates, conditions assigned, etc.). This ensures all participants see changes immediately without requiring page refreshes.

🏗️ SignalR Architecture

System Overview

The SignalR implementation follows a hub-based publish-subscribe pattern where campaigns act as groups. When an event occurs in a campaign, the server broadcasts to all connected users in that campaign group.

⚡ Real-Time Events

Event Types Overview

The application broadcasts the following real-time events through SignalR:

  • CardVisibilityChanged
    Triggered when a location, cast, or sublocation instance visibility toggles. Tells all clients which card is now visible/hidden to players.
  • BulkCardVisibilityChanged
    Triggered when multiple sublocation casts or location sublocations visibility changes at once. Used for parent-child visibility toggles.
  • SecretRevealed
    Triggered when DM reveals a secret in the campaign. Broadcasts which secret was revealed and to whom. Players' UI shows secret immediately.
  • SecretResealed
    Triggered when DM un-reveals a secret. Removes secret visibility from player UIs.
  • SecretDelivered
    Triggered when DM delivers a secret directly to a player card. Notifies all clients that a specific player received a secret.
  • SecretShared
    Triggered when a player shares a delivered secret with party members or other players.
  • ConditionAssigned
    Triggered when DM assigns a condition (status effect) to a player card. Party sees condition applied immediately.
  • ConditionRemoved
    Triggered when DM removes a condition from a player card. Party sees condition disappear immediately.
  • TimeOfDayUpdated
    Triggered when DM creates or updates the campaign's time of day system with day length and slices.
  • TimeCursorMoved
    Triggered when DM moves the time cursor (progress through the day). Shows all clients the current time position.
  • PlayerNotesUpdated
    Triggered when DM updates player-visible notes for a time slice. Players see notes for current time period.
  • DmNotesUpdated
    Triggered when DM updates DM-only notes for a time slice. Other DMs in campaign see the update.
  • NoteUpdated
    Triggered when notes are updated on campaign entities (general purpose note update event).

Workflow 1: Visibility Toggle

When a DM toggles visibility of an NPC card to hide it from players:

DM Hides Cast from Players
This workflow demonstrates how visibility changes propagate in real-time to all connected users.
User Action:
DM clicks hide button on NPC card in Campaign UI
Frontend:
HTTP PATCH /campaigns/{id}/casts/{castInstanceId}/visibility
Request Body:
{ isVisibleToPlayers: false }
Backend Handler:
CampaignsController.UpdateCastInstanceVisibility()
Database Update:
UpdateCastInstanceVisibilityCommand updates visibility flag
SignalR Broadcast:
hubContext.Clients.Group(campaignId).SendAsync("CardVisibilityChanged", event)
Event Payload:
{ campaignId, instanceId, cardType: "cast", isVisible: false }
Frontend Listener:
connection.on('CardVisibilityChanged', (event) => cardVisibilityChanged.set(event))
Signal Update:
Angular signal triggers component re-render
Result:
All clients instantly see NPC card hidden from player views

Workflow 4: Time Cursor Move

When a DM moves the time of day cursor to advance campaign time:

DM Advances Campaign Time
The time of day system allows DMs to track in-game time. Moving the cursor updates for all connected clients.
User Action:
DM drags time cursor on time-of-day bar from 25% to 60%
Frontend:
HTTP PATCH /campaigns/{campaignId}/time-of-day/cursor
Request Body:
{ positionPercent: 60 }
Backend Handler:

1. Signal updated 2. Time bar cursor animated to 60% 3. Current time slice determined 4. Player notes updated
Result:
All connected players' screens show time advanced to same position

💻 Implementation Details

Backend: IHubContext Injection

Controllers inject IHubContext<CampaignHub> to broadcast messages:

public class CampaignsController(
    ...
    IHubContext<CampaignHub> hubContext) : ControllerBase
{
    [HttpPatch("{id}/casts/{instanceId}/visibility")]
    public async Task<IActionResult> UpdateCastInstanceVisibility(
        Guid id, Guid instanceId, [FromBody] UpdateCastInstanceVisibilityRequest request)
    {
        // Update database
        await updateCastInstanceVisibilityCommand.HandleAsync(...);
        
        // Broadcast to all users in campaign group
        await hubContext.Clients.Group(id.ToString())
            .SendAsync("CardVisibilityChanged", new CardVisibilityChangedEvent
            {
                CampaignId = id,
                InstanceId = instanceId,
                CardType = "cast",
                IsVisible = request.IsVisibleToPlayers,
            });
        
        return NoContent();
    }
}

Backend: Hub Configuration

The hub is configured in Program.cs to handle authorization and setup:

// In Program.cs
services.AddSignalR();

app.MapHub<CampaignHub>("/hubs/campaign");

// CampaignHub.cs
[Authorize]  // Requires JWT authentication
public class CampaignHub : Hub
{
    // Client calls this to subscribe to a campaign
    public async Task JoinCampaign(string campaignId) =>
        await Groups.AddToGroupAsync(Context.ConnectionId, campaignId);
    
    // Client calls this when leaving a campaign
    public async Task LeaveCampaign(string campaignId) =>
        await Groups.RemoveFromGroupAsync(Context.ConnectionId, campaignId);
}

Frontend: Service Setup

The CampaignHubService handles all client-side SignalR logic:

@Injectable({ providedIn: 'root' })
export class CampaignHubService {
    private connection: signalR.HubConnection | null = null;
    
    // Signals for different event types
    readonly secretRevealed = signal<SecretRevealedEvent | null>(null);
    readonly cardVisibilityChanged = signal<CardVisibilityChangedEvent | null>(null);
    readonly timeCursorMoved = signal<TimeCursorMovedEvent | null>(null);
    readonly isConnected = signal(false);
    
    async connect(token: string): Promise<void> {
        this.connection = new signalR.HubConnectionBuilder()
            .withUrl(`${environment.apiUrl}/hubs/campaign`, {
                accessTokenFactory: () => token,  // Inject JWT
            })
            .withAutomaticReconnect()  // Auto-reconnect on disconnect
            .build();
        
        // Register all event listeners
        this.connection.on('CardVisibilityChanged', (event: CardVisibilityChangedEvent) => {
            this.cardVisibilityChanged.set(event);
        });
        
        this.connection.on('SecretRevealed', (event: SecretRevealedEvent) => {
            this.secretRevealed.set(event);
        });
        
        // ... more listeners ...
        
        this.connection.onclose(() => this.isConnected.set(false));
        this.connection.onreconnected(() => this.isConnected.set(true));
        
        await this.connection.start();
        this.isConnected.set(true);
    }
    
    async joinCampaign(campaignId: string): Promise<void> {
        await this.connection?.invoke('JoinCampaign', campaignId);
    }
    
    async leaveCampaign(campaignId: string): Promise<void> {
        await this.connection?.invoke('LeaveCampaign', campaignId);
    }
}

Performance & Optimization

Aspect Implementation Benefit
Automatic Reconnection withAutomaticReconnect() configured on hub builder Users don't miss events even if network hiccups occur temporarily
Group-Based Broadcasting Clients only receive messages for campaigns they're in Reduces network traffic and server load for large deployments
Angular Signals Events update reactive signals instead of manual DOM manipulation Automatic change detection and minimal re-renders
JWT Authentication Token passed via accessTokenFactory on connection Secure connection without exposing credentials
Event Filtering Each event type has specific listeners, no universal message handler Cleaner code and faster event processing

Error Handling & Edge Cases

⚠️ Connection Lost: If a user's internet drops during a campaign, they won't receive events while disconnected. SignalR automatically reconnects when connection restores. Components should handle the possibility that UI state may be stale and refresh data when reconnecting.
📝 Multiple Tabs: If a user opens the same campaign in multiple browser tabs, each tab gets its own WebSocket connection. Both receive the same events. This could cause duplicate effect executions - components should be idempotent.
✓ Role-Based Visibility: SignalR broadcasts events to all users in a campaign group regardless of role. Component logic filters what each role sees (players don't see DM-only secrets even if received).
📝 Event Payload Size: Keep event payloads small. Send only necessary IDs and changed flags. Full data fetching happens separately if needed.
